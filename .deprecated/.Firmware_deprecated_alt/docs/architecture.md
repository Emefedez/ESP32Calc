# ESP32-Calc Firmware Architecture

## Folder Structure

```
Firmware/src/
├── main.cpp                  # Entry point: FreeRTOS tasks, boot sequence

│
├── core/                     # Application logic

│
├── screens/                  # Screen implementations (one file per mode)
│   ├── menu_system.h/.cpp    # Screen stack + mode-selection menu
```

## Screen System (core/screen.h)

Every UI mode is a `screen_t` value object. The struct lives in its own header
so screens don't depend on the menu system:

```c
// core/screen.h
typedef struct screen screen_t;

struct screen {
    const char *title;                     // Displayed in header bar
    void (*on_enter)(screen_t *self);      // Called when screen becomes active
    void (*on_draw)(screen_t *self);       // Render pixels to framebuffer
    void (*on_key)(screen_t *self, key_code_t key);  // Handle key event
    void *user_data;                       // Per-screen state (or NULL)
};
```

Each screen lives in its own `.cpp` + `.h` under `screens/`. The header exposes
only a factory function:

```c
// calc_screen.h
#include "core/screen.h"
screen_t *screen_calc_create(void);
```

The `.cpp` defines a `static screen_t` and returns its address. Since the
`screen_t` has no `id` or reference-counting, each mode has a single instance.

## Menu System (screens/menu_system.h)

A **screen stack** (max depth 8). The mode menu lives at depth 1 and is never
popped (guard in `menu_pop()`).

### API

| Function | Effect |
|----------|--------|
| `menu_init()` | Reset stack, push mode menu (depth 1) |
| `menu_push(s)` | Push `s` on top → calls `s->on_enter(s)` |
| `menu_pop()` | Remove top screen; call `on_enter` on the new top |
| `menu_pop_to_root()` | Pop until only depth 1 remains |
| `menu_current()` | Return active screen |
| `menu_draw()` | Call `on_draw` of active screen |
| `menu_handle_key(k)` | Forward key to active screen's `on_key` |

### Mode Menu

The mode menu is built-in. It lists entries from a static `modes[]` array:

```c
static mode_entry_t modes[] = {
    { "Calculator", screen_calc_create()     },
    { "Converter",  NULL                     },  // TODO
    { "Settings",   screen_settings_create() },
    { "Battery",    screen_battery_create()  },
    { "About",      screen_about_create()    },
};
```

Navigation: UP/DOWN to select, EQUALS to enter a mode.
From any sub-screen, **MODE** jumps back to the mode menu
(`menu_pop_to_root()`). AC returns one level, AC on empty calculator
buffer also returns to the mode menu.

## Adding a New Screen

1. Create `screens/my_screen.h`:
   ```c
   #ifndef MY_SCREEN_H
   #define MY_SCREEN_H
   #include "core/screen.h"
   screen_t *screen_my_create(void);
   #endif
   ```

2. Create `screens/my_screen.cpp`:
   ```c
   #include "my_screen.h"
   #include "display/gui.h"
   #include "menu_system.h"

   static void draw(screen_t *self) {
       (void)self;
       gui_clear(GUI_COLOR_WHITE);
       gui_draw_header(self->title, true);
       gui_text(4, 30, "Hello from my screen", GUI_FONT_SMALL, GUI_COLOR_BLACK);
   }

   static void key(screen_t *self, key_code_t k) {
       (void)self;
       if (k == KEY_AC) menu_pop();
   }

   static screen_t screen = {
       .title    = "My Screen",
       .on_draw  = draw,
       .on_key   = key,
   };

   screen_t *screen_my_create(void) { return &screen; }
   ```
   Include `"menu_system.h"` only if you call `menu_pop()` / `menu_push()`.

3. Add an entry in `menu_system.cpp`'s `modes[]` array.

## Settings (core/settings.h/.cpp)

A global struct `g_settings` holds runtime configuration:

```c
typedef struct {
    angle_mode_t  angle_mode;   // ANGLE_DEG or ANGLE_RAD
} settings_t;

extern settings_t g_settings;

void settings_init(void);
void settings_toggle_angle(void);
```

- `settings_init()` is called once in `gui_task()` before the mode menu appears.
- The **Settings screen** lets the user navigate with UP/DOWN and toggle the
  selected setting with CALC.
- The **Calculator screen** reads `g_settings.angle_mode` and displays the
  current mode (`DEG`/`RAD`) in the top-right of the expression area.
- When trig functions (`calc_sin`, `calc_cos`, `calc_tan`) are called, they
  accept an explicit `angle_mode_t` parameter. Callers pass
  `g_settings.angle_mode`.

## Task / Core Layout

ESP32-S3 has two Xtensa LX7 cores:

| Core | Task | Priority | Role |
|------|------|----------|------|
| 0 (pro) | `keypad_task` | 2 | Matrix scan, key queue writes, I/O |
| 1 (app) | `gui_task` | 1 | Display refresh, menu navigation, battery |

**Why separate cores?** E-paper partial refresh holds BUSY for ~300 ms with a
tight GPIO-polling loop. If keypad were on the same core it would be starved
during this window. By running keypad on core 0, keys are scanned and queued
concurrently — the GUI picks them up immediately after the refresh finishes.

## Boot Display Sequence

The boot flow in `gui_task()` (`main.cpp:27-49`):

```
gui_init()           → HW reset, SW reset, register init (no refresh)
battery_init()       → ADC1_CH2 ready
settings_init()      → defaults: DEG, partial
menu_init()          → push mode menu (depth 1)
menu_draw()          → render mode menu to framebuffer
gui_end_ex(FULL)     → one full e-paper refresh (mode menu visible)
drain queue          → discard any stale keypresses during init
└── main loop
    wait for key (50 ms tick)
    loop: drain ALL queued keys → menu_handle_key each
    menu_draw()
    gui_end_ex(PARTIAL)  → one refresh for the batch
```

**Key properties:**
- No splash screen — mode menu appears immediately after one FULL refresh.
- Queue drain prevents stale keypresses from triggering an unwanted PARTIAL
  refresh right after the FULL refresh.
- **Keypress batching**: all keys queued during the previous refresh are
  processed before a single draw+refresh. Rapid navigation presses (e.g. 3× UP)
  finish in ~300 ms total instead of ~900 ms.
- All subsequent refreshes are PARTIAL (fast, low ghosting on SSD1680).

## Header Drawing (display/gui.h)

`gui_draw_header(title, show_back)` draws a 20 px black bar at the top:

```
┌─────────────────────────────────────────────────────┐
│ title text     [AC] back                  battery%  │
├─────────────────────────────────────────────────────┤
```

- `title` is drawn at x=4 (white text on black).
- When `show_back` is true, `[AC] back` is drawn at x=206 (white, ~54 px wide).
- Battery percentage from `battery_draw_indicator()` is drawn at x=268.
- 24 px gap between `[AC] back` and the battery indicator guarantees no overlap
  on the 296 px wide display.

The mode menu draws its own 24 px tall header (taller, to distinguish it from
sub-screens) directly in `menu_draw_fn()`.

## Battery Sensing via SENSE Pin

### Hardware

- **Pin**: GPIO3 (labeled `SENSE` in Wokwi, connected via `wokwi-header`)
- **ADC channel**: ADC1_CH2 on ESP32-S3
- **Expected input**: 0–3.3 V analog (from a voltage divider on the battery)

### Software

`hal/battery.h` exposes these functions:

- `battery_init()` — configures ADC1_CH2 with 12-bit width and 11 dB
  attenuation (0–3.3 V range), seeds the cache
- `battery_read()` — returns a `battery_reading_t` with:
  - `voltage` (float): battery voltage, scaled by `divider_ratio`
  - `percent` (uint8_t): 0–100 estimate (3.0–4.2 V LiPo range)
  Averages 4 ADC samples to reduce noise.
- `battery_read_cached()` — same as `battery_read()` but re-reads ADC at most
  once per 2 seconds **or** every 10 read accesses, whichever comes first;
  returns the cached value between reads
- `battery_force_refresh()` — forces the next `battery_read_cached()` call to
  re-read the ADC. Called automatically from `menu_push()` on mode changes.
- `battery_is_low()` — returns true when voltage < 3.3 V (uses cached)

**The cache + access counter prevent an ADC read on every frame.** All drawing
code (`battery_draw_indicator`, `gui_draw_header`) calls `battery_read_cached()`.

The default `divider_ratio` is 2.0 (assumes a 100k+100k voltage divider).
**Adjust `battery_read()` in `battery.cpp` to match your hardware divider.**

### Adding a Voltage Divider

For a LiPo battery (3.7–4.2 V):

```
BAT+ ──┬─ 100kΩ ──┬── SENSE (GPIO3)
       │          │
      BAT-      100kΩ
                  │
                 GND
```

The divider ratio = (R1 + R2) / R2 = (100k + 100k) / 100k = 2.0

If you use different resistors, update `divider_ratio` in `battery_read()`.

### Wokwi Simulation

In Wokwi, the SENSE header is unconnected by default. To simulate battery
voltage, add a potentiometer to the schematic (`diagram.json`):

```json
{
  "type": "wokwi-potentiometer",
  "id": "bat_sim",
  "attrs": { "value": "3.7" }
}
```

And connect it:

```json
["bat_sim:WIPER", "sense:1", "orange", []]
```

Then adjust the potentiometer value in the Wokwi UI to change the simulated
battery voltage.

### Modifier Keys (ALPHA, SHIFT)

The keypad scanner consumes ALPHA and SHIFT internally — they are never
forwarded to the screen handlers. Instead, call `keypad_alpha_active()` /
`keypad_shift_active()` to query the current state:

- **ALPHA** is a toggle: press once to enable, press again to disable.
- **SHIFT** is auto-release: it deactivates after the next non-modifier key.

The calculator screen displays `ALPHA` or `SHIFT` below the expression area
when the respective modifier is active.

## Raylib Integration

The firmware deliberately stays in C++/Arduino for the first hardware baseline.
Rust has a good `weact-studio-epd` driver, but Wi-Fi, SD SPI, and a raylib-style
software-rendered GUI are lower risk in the ESP32 Arduino/ESP-IDF ecosystem.
The display backend uses the Waveshare `epd2in9_V2` command sequence adapted
locally for the WeAct/Waveshare 2.9" B/W module. The GUI still renders through
`GFXcanvas1`, but the flush path sends SSD1680 commands directly so the project
does not depend on a GxEPD2 panel-class guess.

For GUI compositing with raylib later:

1. Render your scene into a `RenderTexture` / `Image`
2. Extract the raw RGB565 pixel data
3. Call `gui_from_rgb565(pixels)` to convert to 1-bit framebuffer
4. Call `gui_end_ex(GUI_REFRESH_PARTIAL)` to flip to the display

`gui_from_rgb565()` uses a luma threshold of 32768 (50% of 16-bit max). Pixels
with luma < 32768 become black; the rest become white. Pixel layout is row-major
`[y * GUI_WIDTH + x]` in native RGB565 byte order.

## Display Backend (display/gui.cpp)

`gui.cpp` owns a 296×128 `GFXcanvas1`, rotates it into the panel's raw
128×296 memory layout, and sends a 4736 byte 1-bit frame to RAM commands
`0x24` (current) and `0x26` (previous). White pixels are sent as `1`, black
pixels as `0`, matching the Waveshare driver convention.

The Wokwi diagram wires `epd:VCC` directly to 3V3. The real hardware environment
can opt into `PIN_EPD_PWR` with `EPD_ENABLE_PWR_PIN`; if the PCB uses a P-MOSFET
high-side switch, set `EPD_PWR_ACTIVE_LEVEL=LOW`.

The first boot draw uses a full update. Key-driven updates write the full frame
and run the Waveshare partial LUT; after 32 partial updates, the backend forces
one full refresh to keep ghosting under control.

The reset low pulse defaults to 2 ms (`EPD_RESET_LOW_MS`) to avoid BUSY lockups
on boards with reset helper circuitry. The backend logs `[EPD] busy timeout ...`
if the panel keeps BUSY high for more than 6 seconds.

Only the display backend should write SSD1680 registers directly. If the display
stalls at BUSY during this baseline, suspect pin mapping, power polarity, reset
pulse length, or a panel variant mismatch before changing waveform bytes.

### Refresh Modes

| Mode | When used | Flash | Ghosting |
|------|-----------|-------|----------|
| `GUI_REFRESH_FULL` | Boot | Visible flash | None |
| `GUI_REFRESH_PARTIAL` | All key-driven updates | Faint | Minimal |
